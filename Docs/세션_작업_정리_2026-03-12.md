# SNN 연구 세션 작업 정리 (2026-03-12)

## 프로젝트 개요

모바일 6축 IMU 센서(가속도 3축 + 자이로 3축) 데이터로 4가지 제스처(swipe_up, swipe_down, flick_up, flick_down)를 분류하는 **Spiking Neural Network(SNN)** 모델 개발 프로젝트.

기존 CNN 모델과의 공정한 비교를 위해 채널 가변형(Channel-Variant) SNN 모델을 4단계 파이프라인으로 구축하고, 안드로이드 On-Device 실측 테스트까지 수행하였다.

---

## 1. 모델 아키텍처

### 공통 사항
- **입력**: 6축 IMU → 양수/음수 분리 → 12채널 → Rate Coding (z-score → sigmoid → Bernoulli)
- **Conv1d 기반**: 모든 모델이 1D 컨볼루션 사용
- **kernel_size = 9 고정** (CNN, SNN 모두 동일)
- **채널 가변형 (Channel Variant)**:

| Variant | Conv1 채널 | Conv2 채널 |
|---------|:--:|:--:|
| smallest | 16 | 32 |
| small | 24 | 48 |
| medium | 32 | 64 |
| large | 40 | 80 |
| largest | 48 | 96 |

### SNN 4단계 파이프라인

```
Phase 1: Teacher SNN (Soft LIF) — 연속적 membrane potential, 미분 가능
Phase 2: Student KD (Hard LIF + STE) — 바이너리 스파이크, Knowledge Distillation
Phase 3: Sparse Student — Learnable threshold + Firing Rate Regularization
Phase 4: QSparse (INT8 Static Quantization) — Pure Spike Forward Only
```

### CNN / QCNN
- **CNN**: Conv1d(FP32) + GroupNorm + ReLU + Dropout + FC
- **QCNN Static**: Conv1d(INT8) + BatchNorm → fuse_modules → Static Quantization (qnnpack)

### 2-3. Teacher & Student PTL 파일 생성 (70개)

**문제**: models_channel_variant/ 밑에 Teacher, Student 모델이 .pt만 있고 .ptl이 없었음

**원인**:
- Teacher: `LIFNode`의 `buffer=None` 초기화가 TorchScript와 비호환
- Student: `_HardSpikeSTE` 커스텀 autograd가 TorchScript 직렬화 불가

**해결**: `export_teacher_student_ptl.py` 작성
- `ExportableLIFNode`: None 대신 `torch.zeros(1)`로 초기화
- `ExportableHardLIFNode`: 커스텀 autograd 제거, `(v >= threshold).float()` 사용 (추론 시 STE 불필요)
- 기존 .pt 가중치 로드 → Exportable 모델에 매핑 → TorchScript trace → PTL export

**결과**: Teacher 35개 + Student 35개 = 70개 PTL 생성 (전부 성공)

### 2-4. QCNN 정확도 개선 방안 분석

현재 QCNN은 Post-Training Static Quantization(PTQ)을 사용. 개선 방안:

| 순위 | 방법 | 난이도 | 예상 효과 | 재학습 필요 |
|:--:|------|:--:|:--:|:--:|
| 1 | Observer + Per-Channel 변경 | 하 | +0.5~1.5%p | X |
| 2 | QAT (Quantization-Aware Training) | 중 | +1~3%p | O |
| 3 | BN 통일 후 재학습 | 중 | +0.5~1%p | O |
| 4 | 데이터 증강 | 중 | +1~2%p | O |
| 5 | 3층 Conv / Residual | 중 | +1~2%p | O |
| 6 | SNN→CNN KD | 상 | +1~3%p | O |

### 2-5. QSparse vs QCNN 정확도 비교 분석

각 variant별 QCNN 대비 정확도 차이가 가장 큰 QSparse 모델:

| Variant | QCNN Acc | Best QSparse Acc | 차이 | 모델 | PTL |
|---------|:--:|:--:|:--:|------|:--:|
| smallest | 0.9586 | **0.9842** | +2.56%p | T=5 FR5% | 29.8KB |
| small | 0.9645 | **0.9842** | +1.97%p | T=10 FR10% | 46.7KB |
| medium | 0.9744 | **0.9901** | +1.57%p | T=3 FR10% | 41.4KB |

| largest | 0.9763 | **0.9882** | +1.19%p | T=5 FR5% | 70.4KB |

총 65개 QSparse 모델이 QCNN보다 높은 정확도 (PC 시뮬레이션 기준)

### 2-6. 평가 정확도 편차 문제 발견

**문제**: QSparse가 Student KD보다 높은 정확도를 보이는 사례 발견 (파이프라인 상 불가능)

**원인**: 데이터셋의 `__getitem__`에서 매 평가마다 Bernoulli 샘플링으로 다른 스파이크 생성
```python
spike = (np.random.rand(C, N, self.T) < prob[:, :, None]).astype(np.float32)
```

**해결 방안**: 평가 시 랜덤 시드 고정 또는 다수 평가 후 평균

### 2-7. SNN 전용 프로세서 호환성 확인

**결론**: PTL 파일은 뉴로모픽 하드웨어에서 바로 사용 불가

| 항목 | PTL (현재) | 뉴로모픽 HW |
|------|:--:|:--:|
| 대상 | 일반 CPU/GPU | Loihi, Akida, Speck 등 |
| 포맷 | TorchScript Lite | 각 HW 전용 포맷 |
| 연산 | MAC (행렬 곱) | AC (조건부 덧셈) |
| 변환 | - | 가중치+LIF 파라미터 추출 → SDK 변환 |

뉴로모픽 HW 배포 시 SynSense(Sinabs), BrainChip(Akida), Intel(Lava) SDK로 변환 필요.
우리 모델이 이미 바이너리 스파이크 + LIF 구조이므로 변환 자체는 비교적 수월.

## 5. 핵심 결론

1. **PC 시뮬레이션**: QSparse SNN이 QCNN보다 정확도 우세 (65개 모델이 QCNN 초과)
2. **On-Device 실측**: 일반 프로세서에서는 QCNN이 전 지표 우세 (정확도, 지연시간, 에너지)
3. **SNN의 진정한 이점**: 뉴로모픽 하드웨어에서 AC 연산 + 스파이크 스킵으로 실현
4. **평가 편차 문제**: Bernoulli 샘플링으로 인한 평가 시 정확도 편차 → 시드 고정 필요
5. **최적화 효과**: Teacher → EE QSparse로 지연시간 89.8%↓, 에너지 76.3%↓ 달성
